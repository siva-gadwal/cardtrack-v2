📦 What Is This App?

CardTrack v2 is a credit card expense tracker — completely client-side. There is no server. No database. No login API. Everything runs in your browser and is stored locally on device.

This is important: if you clear browser storage, all data is gone. There is no cloud sync.

✓ What Exists

  • React frontend (SPA)
  • Browser LocalStorage (data)
  • Service Worker (offline)
  • PWA manifest (installable)
  • Netlify (static hosting)

✗ What Does NOT Exist

  • No backend server
  • No REST / GraphQL API
  • No database (SQL/NoSQL)
  • No user accounts
  • No cloud storage

🔄 Full Data Flow — Adding a Transaction

Here is exactly what happens when a user logs an expense, step by step:

1

User Action

Taps "Add Transaction." React updates screen state to "addTxn". No page load — it's a SPA.

2

Form Input

User fills amount, picks card, category, description, date. Each keystroke updates local component state via useState.

3

Validation

On submit: check amount is a number, card is selected. If not — show inline error. No network call needed.

4

Threshold Check (Notifications)

Before saving, app calculates new utilization %: (spent + newAmount) / cardLimit × 100. If crosses 75% or 90% for first time → triggers browser Notification API.

5

ID Generation

New transaction gets a unique ID: Math.random().toString(36).slice(2,10) — 8-char random alphanumeric string. No UUID library needed.

6

React State Update

setData(d => ({...d, transactions: [...d.transactions, newTxn]})) — immutable update appends transaction to in-memory array. React schedules re-render.

7

Side Effect → Persistence

useEffect([data]) fires on every data change. Calls localStorage.setItem("cardtrack_v2", JSON.stringify(data)). Data is now written to disk.

8

UI Re-renders

React diffs the virtual DOM, updates only changed elements. Spending bar, category totals, transaction list all reflect new data. Toast "Transaction saved" appears briefly.

9

Next App Load (Browser Restart)

On mount, loadData() calls JSON.parse(localStorage.getItem("cardtrack_v2")). State is hydrated. App resumes exactly where it was.

🗄 Data Storage Schema

All data lives in one JSON blob in localStorage under key cardtrack_v2.

localStorage["cardtrack_v2"] = { "cards": [ { "id": "x3f9ab12", // random 8-char ID "name": "Chase Sapphire Preferred", "limit": 15000, // dollars "color": "#0a2a5c", "billingDay": 15 // day of month cycle resets } ], "transactions": [ { "id": "k8m2p3n7", "cardId": "x3f9ab12", // FK to cards[].id "amount": 125.50, "category": "Dining", "description": "Restaurant ABC", "date": "2026-04-25" // ISO 8601 } ] }

Relationships are manual: transaction.cardId links to card.id. No foreign keys. Joins done in JS with .find().

Storage limit: ~5–10 MB (browser-enforced). Thousands of transactions fit easily.

🔐 Authentication

There is no server-side auth. The "PIN lock" is purely client-side UI protection.

1

First Run — Setup

User sets 4-digit PIN → PIN is stored as plain text in localStorage["cardtrack_pin"].

2

Subsequent Opens

App reads PIN from localStorage. Shows lock screen. User types 4 digits. enteredPin === storedPin → unlock. No server round-trip.

3

PIN Reset

Button clears mode → setup flow restarts. Old PIN overwritten. No recovery mechanism.

⚠ IT Note: PIN is plain text in browser storage. Anyone with DevTools access (F12 → Application → LocalStorage) can read it. This is app-level screen lock, not cryptographic security. Do not store sensitive financial credentials in this app.

📶 Offline Support (Service Worker)

After the first page load, CardTrack works with no internet. Here's why:

// sw.js — Network-first, cache-fallback strategy

self.addEventListener("fetch", (e) => {
  if (e.request.method !== "GET") return;

  e.respondWith(
    caches.match(e.request).then((cached) => {
      // Try network first
      const fetchPromise = fetch(e.request)
        .then((response) => {
          if (response?.status === 200) {
            caches.open(CACHE_NAME).then(cache =>
              cache.put(e.request, response.clone())  // update cache
            );
          }
          return response;
        })
        .catch(() => cached);       // offline? use cache

      return cached || fetchPromise;  // instant if cached
    })
  );
});
How it works: Browser registers sw.js as a background thread. It intercepts every GET request. If cached version exists → serve it instantly. In parallel, fetch fresh copy from network and update cache. If offline → cached version serves the app. Data (localStorage) never goes to network, so offline writes work too.

⚡ TL;DR for IT

One sentence: CardTrack is a React single-page app that runs entirely in the browser, stores everything in localStorage as JSON, has no server, no database, and no network traffic beyond loading the app itself.

Stack: React 18 + Vite 5, hosted on Netlify as static files.
Data: ~3 localStorage keys, JSON blobs, max ~5MB.
Auth: 4-digit PIN, plain text, client-side only.
Offline: Service Worker caches all app files after first load.
Install: PWA — can be added to home screen, works like a native app.
Risk surface: Local only. No external data exfil. Physical device access = full data access.
